## 二、逐维度详评

### 2.1 Originality（原创性）：2.5 / 5

#### 加分项

| 项目 | 说明 |
|---|---|
| ATSS 在线学习范式 | 将 MAB 框架应用于证明搜索是相对新颖的。避开了神经方法对大规模预训练数据的依赖，有自己的设计哲学。 |
| CDCL + Craig 插值 + 在线学习的三合一架构 | "良性循环"的概念有说服力——至少在设计层面是完整且有逻辑的。 |
| LEMMA_REUSE 的增量构造方式 | 在证明构建过程中（而非事后）进行 DAG 共享是一个区别于 DRAT/LRAT 等事后压缩方法的设计选择。 |

#### 减分项

| 项目 | 说明 |
|---|---|
| 混合演算本身不算创新 | ND + SC + Resolution 的组合在文献中已被广泛研究（如聚焦证明系统、极化归结等）。论文的演算层面没有引入新的推理规则原理——三条"新规则"可以看作启发式优化而非演算级别的新规则。 |
| ATSS 收敛定理的技术深度有限 | 证明主要依赖加权 Hoeffding + union bound，这是标准的集中不等式套路。非 i.i.d. 的论证方向正确但缺乏严谨性（见技术质量部分）。 |
| Craig 插值的集成 | Pudlák 算法直接使用，无算法层面的改进。集成到在线学习中的 idea 有新颖性，但实现是直截了当的。 |
| GNN-ATSS 实验 | 在 50 个公式的极小 benchmark 上测试，没有展示任何优势，削弱了 GNN 组件的原创性主张。 |

#### 综合评估

论文的原创性在于**架构设计**（三个组件的协同）而非**单项技术**的突破。ATSS 作为核心创新，其 MAB 方法有新颖性但属于应用层面而非方法论层面的创新。综合评价：**2.5/5**——超过"纯工程实现"但未达到"方法论突破"。

---

### 2.2 Significance of Topic（主题重要性）：3.0 / 5

#### 加分项

| 项目 | 说明 |
|---|---|
| 命题证明系统的可信性问题 | 这是一个真实的、正在被学术界和工业界共同关注的问题。Verified SAT（Fleury 2025）、DRAT/LRAT 验证器等近年的工作证明了这一点。 |
| 零预训练证明搜索 | 避免了神经方法的数据依赖，这对于新领域/新逻辑的证明搜索尤其重要——在 Lean/Coq 中，为每种新理论建立训练语料是不现实的。 |
| 形式化验证的实际价值 | de Bruijn 准则是定理证明社区的核心方法论。论文在这一标准上的遵循是正确的。 |

#### 减分项

| 项目 | 说明 |
|---|---|
| 命题逻辑范围的局限 | 命题逻辑是一个已经被"解决"的领域——SAT 求解器在工业问题上已经非常成熟。论文没有清楚地论证为什么需要在命题层级重新发明一个混合系统，而不是直接在更有挑战性的一阶/高阶逻辑中证明价值。 |
| 实验没有展示实际优势 | 如果在命题逻辑这个"简单"领域中都无法展示比 DPLL 更好的性能，读者自然会质疑：这个系统在更复杂的领域中能做什么？ |
| 目标社区的匹配度存疑 | LICS 社区更看重理论深度（当前论文的理论贡献偏浅）；CAV 社区更看重实证强度（实验数据不支持）。 |

#### 综合评估

命题证明系统的可信性是有意义的主题，但论文没有充分论证为什么需要在命题层级（而非一阶或高阶）解决这个问题。实验数据的薄弱进一步削弱了主题的实际重要性。**3.0/5**。

### 2.3 Technical Quality（技术质量）：2.5 / 5

#### 加分项

| 项目 | 说明 |
|---|---|
| 完备性证明正确 | p-模拟 Frege 的三步翻译是正确的标准论证。 |
| 可靠性证明完整 | Rocq 中的形式化涵盖了 17 条 ND 规则，双重语义的桥接引理是扎实的工作。 |
| ATSS 收敛定理 | 推导过程在技术上是正确的（在标准假设下），提供了非平凡的形式化保证。 |
| 代码完整、可运行 | 完整的 Python 实现 + Rocq 形式化是可验证的工程成果。 |

#### 减分项（按严重程度排序）

| 严重程度 | 项目 | 说明 |
|---|---|---|
| **致命** | CDCL 实现质量 | CDCL 在多个 benchmarks 上不如 DPLL 是不应该发生的。DPLL ⊂ CDCL（CDCL 是 DPLL 的超集），CDCL 在任何实例上的性能不应比 DPLL 差。这强烈暗示实现中存在 bug 或严重的参数配置问题。 |
| **严重** | 非 i.i.d. 论证不够严谨 | ATSS 收敛定理的证明依赖于"加权 Hoeffding 不等式不需要独立性"的声称，但标准的加权 Hoeffding 仍需要独立性（或鞅差条件）。论文的"局部平稳"替代条件是合理的直觉，但在技术上需要更严格的处理（如 Azuma-Hoeffding）。 |
| **严重** | 实验规模不足以支持统计结论 | 50 次 trial 在许多实验中不足以得出统计显著结论。8 个实验中有 3 个展示了 NeuroProof 的"失败"。 |
| **中等** | PHP 实验数据可疑 | DPLL 对 PHP_n^{n+1} 的解决时间（13-20秒）与已知的指数级下界矛盾。这需要解释或重新验证。 |
| **中等** | ATSS 效果未被实验支持 | Exp. 4 显示 ATSS 与 noATSS 表现相近。论文在 discussion 中承认但没有解决方案。 |
| **轻微** | GNN-ATSS 实验设计不当 | 50 个小公式不足以展示 GNN 的任何优势——benchmark 太小，GPU 开销占主导。 |

#### 综合评估

理论部分有一定质量但存在技术不严谨之处（非 i.i.d. 论证）。实验部分是论文最薄弱的地方，严重削弱了系统主张的可信度。**2.5/5**——勉强可接受但存在严重缺陷。

### 2.4 Presentation（表达质量）：3.5 / 5

#### 加分项

| 项目 | 说明 |
|---|---|
| 结构清晰 | IEEEtran 格式规范，章节逻辑流畅。Introduction → Background → System → ATSS → Interpolation → Rocq → Experiments → Conclusion 的结构是合理的。 |
| 写作流畅 | 英文表达专业且清晰，无明显语法错误。技术术语使用准确。 |
| 诚实性 | 论文在多处诚实地承认了局限性——CDCL 限制、完备性未形式化、实验规模的不足。这体现了良好的学术态度。 |
| 图表质量 | 对比表（Tab. 5 SOTA 对比）和实验表格清晰易读，排版得当。 |
| 相关工作全面 | 覆盖了四个方向的相关工作，对 DreamProver 等最新工作的讨论是及时的。 |

#### 减分项

| 项目 | 说明 |
|---|---|
| Abstract 略显拥挤 | 包含太多细节（T1-T3 全部列出），使得 abstract 信息密度过高。 |
| 部分结论过于乐观 | "the first integration of Craig interpolation into an online proof-search strategy" 的声称是准确的，但"first"类的声称在论文中出现了多次，容易引起审稿人反感。 |
| PHP 实验的讨论有回避性 | 关于 DPLL 反常数据的讨论过于简短。 |
| 缺少相关的对比讨论 | GNN-ATSS 比 Cosine-ATSS 慢 200 倍但没有提供深入分析。 |

#### 综合评估

写作和结构是论文的最强项。但在实验讨论的深度和某些声称的谨慎性上有改进空间。**3.5/5**。

## 三、综合评估

### 按目标会议的评估

#### LICS 2027

| 维度 | LICS 期望 | 本文表现 | 匹配度 |
|---|---|---|---|
| 理论深度 | 需要深刻的新定理或证明技术 | ATSS 收敛定理有一定深度但技术路线是标准套路 | **偏低** |
| 新颖性 | 需要概念性突破 | 架构设计有新意但缺乏方法论创新 | **偏低** |
| 与 LICS 传统的关联 | 与证明复杂性、逻辑的深层联系 | 与 Cook-Reckhow、p-模拟等有联系，但属于应用层面 | **中等** |

**LICS 录取判断：Weak Reject (40%)**
- 理论贡献的深度不足以支撑 LICS 的标准
- 如果强化 ATSS 收敛定理的证明（修复非 i.i.d. 论证）、增加更深刻的理论结果（如与证明复杂性的更紧密联系），录取可能性会提高

#### CAV 2027

| 维度 | CAV 期望 | 本文表现 | 匹配度 |
|---|---|---|---|
| 系统实现 | 需要可验证的系统 | 完整的 Python 实现 + Rocq 形式化 | **良好** |
| 实验验证 | 需要充分的实验证据 | 实验规模不足，核心系统性能不佳 | **严重不足** |
| 实用性 | 需要展示实际价值 | 在命题逻辑层面未展示实际优势 | **不足** |

**CAV 录取判断：Weak Reject (45%)**
- 实现和形式化是加分项
- 但实验数据严重不足——CAV 审稿人会对 CDCL 不如 DPLL 的现象提出严厉质疑
- 如果修复 CDCL 实现 + 补充大规模实验 + 展示 ATSS 优势 → 录取可能性将显著提升

## 五、最终建议

**当前状态**：论文在架构设计、写作质量和形式化验证方面有明确的优点，但**实验数据不支持核心主张**——这是最需要解决的根本问题。CDCL 实现比 DPLL 差的现象是审稿人最容易攻击的点。

**如果投稿前无法完成 P0 修复**：建议降低目标会议层级（如 IJCAR、TACAS），或大幅修改论文论述以更诚实地呈现当前系统的实验状态，将重点从"性能"转向"架构可行性论证"。

**如果完成 P0 修复**：CAV 2027 是比 LICS 2027 更合适的目标，因为 CAV 更看重有实现的系统。LICS 需要更深刻的理论贡献，而 ATSS 收敛定理在理论深度上可能不足以达到 LICS 标准。
